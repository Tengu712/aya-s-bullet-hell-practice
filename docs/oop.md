# Object Oriented Programming

## Outline

本開発は、勉強も兼ねて過剰に、オブジェクト指向プログラミングパラダイムに従っている。

本ドキュメントは、私なりのオブジェクト指向開発における注意事項を記している。

## Object Decomposition

オブジェクトとは、データあるいはロジックあるいはその組である。
とはいえ、実際TypeScriptでコーディングしてみると、単なるデータはユーザ定義型として扱うし、単なるロジックは関数型データとして扱うため、データとロジックの組であることが多い。

オブジェクトは以下の思考手順で切り出す。切り出して以降、インターフェースを介してプログラムを進める。

1. 「プログラムのどの部分を切り替えやすくするか」を考える
1. その部分をインターフェースに置き換える
1. 実際的なプログラムを実装する

例えば、~~今回実装する予定はないが~~、プレイヤーキャラクターの動作をプレイヤーの入力だけでなくリプレイデータからも行えるようにするかもしれないとする。

1. プレイヤーキャラクターの「動き」を切り替えたい
1. 「動き」を返すロジックを含む`IPlayerMovement`インターフェースを定義する
1. `PlayerInput`クラスと`ReplayInput`クラスを定義する
1. プレイヤーキャラクターはインターフェースを介して「動き」を得る
1. プレイヤーの外部から具象クラスのインスタンスを与える

ポリモーフィズム主義的な設計方針かもしれないが……。

## Always Define Interfaces

実装が一つしか想定されないオブジェクトでも、オブジェクトに切り出すべきと判断される限り、必ずインターフェースを定義する。

例えば、本プロジェクトで IRenderer は WebGL2 しか実装していない。
確かに、将来、レンダラーを切り替える機能を付けるとは考えにくい。
なんなら、グローバルスコープに置いて`console.log()`の要領で描画リクエストを行ってもいいかもしれない。

しかし、ひとたび具象オブジェクトの利用を許可すると、雑な設計に「甘え」たくなってしまう。
雑な設計が破綻したとき、依存負債は本領を発揮する。
個人の小規模開発とはいえ、本開発ではこれを良しとしないと決めた。

## Two Domains

疎結合を図るため、領域を Structure と Control に分ける。

- Structure
  - プログラムの構成部品
  - 他のオブジェクトとは必ずインターフェースを介す
  - ユーザ定義型は Structure に属す
- Control
  - プログラムの流れ
  - インスタンスの生成を許し、そのために限り具象オブジェクトの`import`を許す
  - メインクラス、ファクトリクラス、シーンクラスの実装のみが Control に属す
